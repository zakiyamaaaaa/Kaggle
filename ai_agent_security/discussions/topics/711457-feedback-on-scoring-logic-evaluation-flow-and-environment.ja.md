## 投稿

こんにちは。

まず、このコンペを主催してくださった主催者の方々に感謝します。しかし約 10 日間取り組んだ結果、いくつかの問題が明らかになりました。他の参加者が複数のスレッドで指摘している点と重複するものもありますが、多くの懸念に十分な回答が得られていないように見えるため、重複を気にせずここに整理しておきます。

### スコア関数が単純なループを奨励している

現在のスコア関数は、コンペの意図した目的を反映していないようです。具体的には、全く同じ成功した攻撃アクションを繰り返すだけでスコアが単調に増加します。

この挙動により、参加者は「1 秒あたりのスコアを最大化する攻撃」を見つけ、時間切れまで盲目的に繰り返すことを強いられます。これは脆弱性を探索するアルゴリズム手法を提案するという中核目標と一致しません。さらに、replay 環境の正確なレイテンシが参加者にわからないため、タイムアウトしない回数を推測する不要な「チキンレース」が生じます。

これを解決するには、全く同じ finding は 1 回だけ報酬を与えるようにスコアリングを設計すべきです。

### Public と Private リーダーボードの相関が弱い

Public LB への過学習を防ぐ重要性は理解していますが、現状の問題は public guardrail と private guardrail が完全に別個の単一エンティティのように見えることです。

そのため public スコアは、アルゴリズムの有効性の妥当な代理指標として機能しません。具体的には、public 環境で使われる現在の OptimalGuardrail は 4 種類の攻撃タイプのうち 2 つをほぼ無力化し、`http.post` で固定の埋め込み文字列を送るだけの戦略が過度に強くなります。これにより探索アルゴリズムの設計が非常に困難になります。public LB では攻撃タイプの半分を検証できず、public スコアは賢い探索アルゴリズムを報いません。その結果、private guardrail の隠れた性質を推測することが最終順位を大きく左右します。public スコアは開発のための実用的なフィードバックループを提供する必要があります。

### 単一の Private Guardrail での評価は分散が大きい

最終順位は private guardrail 評価で決まりますが、private guardrail を 1 種類だけ使うのは問題です。

このコンペは ideally、多様な guardrail に対する攻撃探索手法の汎化能力を評価すべきです。1 つの guardrail だけで評価すると、最終結果が賭けになります。堅牢で汎化された手法が、その特定の guardrail との相性が悪ければ低スコアになる可能性があります。逆に、汎用性が限られた狭い手法が運良く高順位になることもあります。この高分散は、チームの方法的な実力とリーダーボード順位の相関を大幅に下げます。汎化を正しく測るには、複数の異なる guardrail で評価し集約すべきです。

### 評価フローと時間ベースのサイドチャネルリスク

public と private 環境で独立した探索フェーズが実行されるかどうかについて、依然として曖昧さがあります。

探索フェーズが 1 回だけ実行され、生成された finding が public と private の両 guardrail で評価される場合、private LB は事前生成された finding が private guardrail をどれだけ bypass できるかだけを測り、private guardrail からのフィードバックを使って適応するエージェントの能力を評価できません。

一方、public と private で独立した探索フェーズを走らせる場合、現在のインフラは参加者が private guardrail の性質を推測するために使えるタイミングのサイドチャネルを露出する可能性があります。私の観察では、提出の処理に約 10 時間かかります。これから、public と private の両評価が完全に終わった後にスコアが返ると推測しています。

この設定下では、参加者は提出の処理時間をサイドチャネルとして使い、private guardrail に関する情報を漏洩できます。例えば、private guardrail が `http.post` を禁止しているかどうかを判定するために、探索フェーズで `http.post` を試みます。成功したら、提出は即座に空の finding リストを返して実行を瞬時に終了できます。ブロックされたら、replay に何時間もかかる長い finding リストを返します。private guardrail が `http.post` を許可していれば、提出は数分で 0.0 スコアで終了します。禁止されていれば、評価は何時間も走ります。この応答時間を測ることで、コンペ期間中に private guardrail のポリシーを体系的にマッピングできます。

探索フェーズが 1 回だけの場合でも、同様のタイミングサイドチャネルが存在する可能性があります。replay フェーズで、事前生成された多段 finding に、以前のツール呼び出しの出力を調べるプロンプトを後のターンに含めることができます。そのツール呼び出しが許可されたか拒否されたかによって、エージェントはより短いか長い実行シーケンスに切り替えます。このような finding を繰り返すと、総評価時間の差が増幅されます。

### スコアリング環境での GPU スループットの不一致

ユーザー Kawasaki（@zhtngr）が以前指摘したように、Kaggle Notebook の T4 GPU と実際のスコアリング環境の実行時間には巨大な差があります。

通常のノートブックでは、2,000 件の finding の replay に 9,000 秒はかかりません。しかし実際の提出のほとんどでは、600〜800 件の replay だけで 9,000 秒の budget をほぼ使い切ります。

主催者はコンペで GPU が使われていると述べていますが、観測しているスループットは Kaggle Notebook の T4 GPU で期待されるものと一致しません。バックエンドのリソース割り当てと潜在的なボトルネックについて、明確な説明が必要です。

@owenvallis さんのこれらの点についての見解をぜひ聞きたいです。改めてこの挑戦的なコンペを主催してくださりありがとうございます。これらの観察が皆さんの体験改善の助けになれば幸いです。
