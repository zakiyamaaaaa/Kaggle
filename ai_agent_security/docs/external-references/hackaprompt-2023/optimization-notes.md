# HackAPrompt 最適化メモ

`gstaff/hackaprompt-solutions` は、最終解だけでなく `v1` から複数版の提出 JSON が残っている点が有用。履歴を見ると、成功する長めのプロンプトを見つけたあと、同じ成功条件を保ちながら不要な文字を削っている。

## 文字数の推移

手元で raw JSON を取得し、各 level の `prompt` 文字数を合計した結果:

- main submissions: `v1` は 824 文字、`v25` は 315 文字。
- flan-only submissions: `v1` は 712 文字、`v5` は 282 文字。
- どちらも非空の level は 9 個で、level 10 は未解決または空。
- level 9 は最終的に 10 文字まで圧縮されている。

この変化は、最初から最短解を狙うのではなく、成功を確保してから縮める流れを示している。

## 観察できる圧縮の方向

### 1. 長い構造から短い命令へ

初期版では `Rules:`、`Sentence:`、区切り線、条件分岐、引用符が多い。後半では、同じ役割を果たす最小のフレーズだけが残っている。

今回の実験では、成功 trace を得たあとに以下を順に削るとよい。

- 前置き説明。
- 例示。
- 丁寧語や自然文の補助語。
- 重複した条件。
- 引用符や区切り記号。
- 1 ターン目と 2 ターン目のどちらかに重複している情報。

### 2. 成功率と短さを分けて扱う

README には、ChatGPT の level 2 と level 7 は不安定で、level 2 の最後の解は成功率が約 10% と書かれている。つまり、最短解は必ずしも高成功率ではない。

今回のコンペでは replay 可能な candidate を返すため、次のように分けて管理する。

- 探索用: 長くても成功率が高いプロンプト。
- 提出候補用: replay で通る範囲まで削ったプロンプト。
- 低成功率枠: 安い試行として複数投入する候補。
- 安定枠: スコアの土台にする候補。

### 3. モデル別に圧縮する

main submission と flan-only submission では、同じ level でも残った語彙が違う。例えば ChatGPT 系では翻訳・条件・整形の文脈が残りやすく、FlanT5 系では `Repeat`、`Echo`、`English` のような直接命令が短く残っている。

今回の実験では、GPT-OSS と Gemma を同じテンプレートで最適化しない。成功候補をモデル別に分け、以下を記録する。

- どのモデルで成功したか。
- 成功時の tool sequence。
- guardrail deny の有無。
- 文字数。
- その候補が他モデルでも replay 成功するか。

### 4. 文字単位の削除探索

README には、level 9 の Flan 解法では既存解から文字を削る自動探索を使ったとある。これは今回の `AttackCandidate.user_messages` にも応用できる。

安全な削除探索の手順:

1. 成功した候補を固定する。
2. 1 文字、1 単語、1 行、1 節の単位で削った候補を生成する。
3. ローカル評価で replay 可能性を確認する。
4. 成功した最短候補を残す。
5. 最短候補だけでなく、少し長いが安定する候補も残す。

### 5. 圧縮対象は文面だけではない

今回のコンペでは、HackAPrompt と違い multi-turn と tool trace がある。そのため、圧縮対象は user message の文字数だけではない。

削る対象:

- ターン数。
- 1 ターンあたりのツール呼び出し数。
- 事前準備に使うファイル読み取りや検索。
- 同じ predicate を発火する重複ルート。
- replay 時に時間を食うがスコア差が小さい候補。

残す対象:

- unique cell を変えるツール列。
- severity が高い predicate を発火する経路。
- public と private の両方で通りそうな一般化された行動文脈。
- 失敗時にも次の候補へ速く進める短い経路。

## 実験ノートへの反映案

各 `experiments/runs/<exp>/notes.md` に、今後は次を追記すると比較しやすい。

- `seed_prompt`: 最初に成功した長めの候補。
- `compressed_prompt`: 提出候補に残した短縮版。
- `removed_parts`: 削っても成功した要素。
- `required_parts`: 削ると失敗した要素。
- `stability`: ローカル試行での成功率。
- `model_split`: GPT-OSS / Gemma の成功差。

この形式にすると、HackAPrompt のように「成功候補の履歴そのもの」を資産化できる。
