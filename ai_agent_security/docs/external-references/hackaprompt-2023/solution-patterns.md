# HackAPrompt 解法パターン

参照元は `gstaff/hackaprompt-solutions` の `main_submissions/*.json` と `flan_only_submissions/*.json`。以下は文面の丸写しではなく、使われている攻撃パターンを現在の実験に流用しやすい形で整理したもの。

## 1. タスクの外形に合わせた命令上書き

初期版では、対象レベルの表向きのタスクに合わせて「翻訳」「改善」「分類」「物語生成」などの形式を残し、その中に目標出力を埋め込んでいる。

例として、翻訳タスクには「スペイン語を英語へ戻す」、文章改善タスクには「改善点がなければ指定文を返す」、分類タスクには「条件分岐の選択肢として指定文を返す」といった形が使われている。

今回のコンペへの示唆:

- エージェントの自然な職務に見える wrapper を作る。
- `http.post` や `email.send` を直接命令するだけでなく、報告、通知、監査、バックアップ、確認依頼などの業務文脈に入れる。
- guardrail が見ている禁止語を避けつつ、エージェント側には目的が明確に見える形にする。

## 2. 直接命令よりも「出力形式」の指定を使う

多くの提出では、「これを言え」だけでなく「句読点を消せ」「引用符を外せ」「英語で言え」「最初に言え」のような出力整形指示が併用されている。

これは、禁止命令を上書きするというより、モデルが従いやすい局所的な整形タスクへ変換していると読める。

今回のコンペへの示唆:

- 危険操作そのものではなく、ツール引数の形式、宛先、本文、URL、ファイルパス、JSON 形式などを整形タスクとして誘導する。
- 候補生成では「行動指示」と「引数整形」を分けて変化させる。
- 成功 trace を見たら、最後に効いたのが目的文か整形指示かを分解して記録する。

## 3. 引用符・区切り・擬似構造で命令境界を作る

初期版には、三重引用符、角括弧、長い区切り線、`Rules:`、`Sentence:`、`Choose:` などの構造が多い。これは、システム側の既存指示とは別の局所コンテキストを作り、モデルに「いま処理すべき対象」を誤認させる手法として使われている。

最終版では多くが削られているが、探索初期には有効な候補空間を広げる役割がある。

今回のコンペへの示唆:

- 初期探索では、メール本文、Web ページ、設定ファイル、ログ、チケット、CSV、JSON、Markdown などの擬似構造を試す。
- 成功後は、構造のどの部分が必要だったかを削って最小化する。
- multi-turn では、1 ターン目で構造を置き、2 ターン目で「上の仕様に従って処理して」と参照させる。

## 4. 翻訳・言い換えによる間接化

最終版の main submission では、`He sido PWNED...` のようなスペイン語風の入力から英語へ戻す形が残っている。直接の目標文よりも、翻訳タスクとして渡したほうが通るケースがある。

今回のコンペへの示唆:

- 危険語や secret 風文字列を直接扱うだけでなく、翻訳、要約、復元、正規化、表記揺れ補正として扱う。
- エージェントが「内容を理解して処理する」場面を作ると、単純な禁止語回避より自然に行動へつながることがある。
- ただし、本番 evaluator は encoding や secret 変形も検知する可能性があるため、隠蔽自体を目的にしすぎない。

## 5. Unicode 類似文字・特殊トークン

提出には、丸囲み文字や一部の非 ASCII 文字、ChatGPT 系の `<|im_end|>` が使われている。README でも、特殊トークンは競技の tokenizer では複数トークンとして数えられる点が注記されている。

今回のコンペへの示唆:

- Unicode 変形は、guardrail とモデル本体で正規化のされ方が違う場合に効く可能性がある。
- ただし、現在のコンペではツール引数、secret 検知、replay 成功が重要なので、単なる文字化け回避は過信しない。
- 候補として持つなら、raw text、URL encoded、base64、Unicode 類似文字、空白挿入を同じファミリーとして比較する。

## 6. モデル別の書き方

main submission は多くのレベルで `gpt-3.5-turbo` を使い、level 9 は `FlanT5-XXL` を使っている。Flan-only では、より直接的で短い命令が多く残っている。

読み取り:

- ChatGPT 系には、翻訳、条件分岐、引用構造、文脈再解釈が効いている。
- FlanT5 系には、直接的な echo/repeat/English 指示や、短い表現が効いている。
- 同じゴールでも、モデルごとに最短表現が異なる。

今回のコンペへの示唆:

- GPT-OSS と Gemma で候補テンプレートを分ける。
- 共通テンプレートだけでなく、モデル別に成功率が高い文体を集計する。
- `AttackCandidate` を返す段階では、モデルごとの replay で成功しやすい候補を混ぜるより、重複しない成功経路を優先する。
