# Dataset EDA 日本語メモ

対象データ:

- `data/raw/train.csv`
- `data/raw/test.csv`
- `data/raw/sample_submission.csv`

作成日: 2026-07-08

## 要約

このデータセットは、科学系の多肢選択問題に対して `A`〜`E` の5択から正答を選ぶ形式です。`train.csv` は 200 問で正解ラベル付き、`test.csv` は 200 問で正解ラベルなし、`sample_submission.csv` は提出形式のサンプルです。

重要な注意点として、ローカルに配布されている `test.csv` は、`train.csv` から `answer` 列だけを除いた内容と完全一致しています。そのため、この `test.csv` を使った検証は隠しテストの代替にはなりません。ローカル検証では `train.csv` を分割するか、外部データで別の validation set を作る必要があります。

## ファイル構造

| ファイル | 行数 | 列 |
|---|---:|---|
| `train.csv` | 200 | `id`, `prompt`, `A`, `B`, `C`, `D`, `E`, `answer` |
| `test.csv` | 200 | `id`, `prompt`, `A`, `B`, `C`, `D`, `E` |
| `sample_submission.csv` | 200 | `id`, `prediction` |

全ファイルで欠損値はありません。`id` は各ファイルで 200 件すべて一意です。

## train/test の関係

`train.csv` と `test.csv` について、`id`, `prompt`, `A`, `B`, `C`, `D`, `E` を比較すると全 200 行が一致します。

これは Kaggle の実運用上の隠しテストとは別に、配布ファイルとして小さな公開サンプルが置かれている状態と考えるべきです。`test.csv` への提出ファイルは作れますが、モデル性能を測る用途には使えません。

## 正解ラベル分布

| ラベル | 件数 | 割合 |
|---|---:|---:|
| A | 37 | 18.5% |
| B | 48 | 24.0% |
| C | 44 | 22.0% |
| D | 38 | 19.0% |
| E | 33 | 16.5% |

完全に均等ではありませんが、極端な偏りではありません。最頻ラベルは `B`、次が `C` です。

固定順序で提出するだけでも、ラベル分布の影響で MAP@3 が少し変わります。

| 固定・単純ルール | train 上の MAP@3 |
|---|---:|
| sample submission 相当: `A B C` | 0.3783 |
| ラベル頻度順: `B C D` | 0.4133 |
| 最良の固定順序: `B C D E A` | 0.4133 |
| 逆順例: `E D C` | 0.3333 |
| ランダム順序の期待値 | 0.3667 |

この程度のスコアは内容を読まなくても出るため、モデルや検索手法の評価では最低限これを上回る必要があります。

## テキスト長の分布

`prompt` は短めで、選択肢のほうが長い構成です。

| 対象 | min | p25 | median | mean | p75 | max |
|---|---:|---:|---:|---:|---:|---:|
| prompt 語数 | 3 | 6.0 | 10.0 | 11.4 | 14.0 | 42 |
| 正解選択肢の語数 | 1 | 16.0 | 27.0 | 28.6 | 40.3 | 111 |
| 不正解選択肢の語数 | 1 | 13.0 | 22.0 | 25.6 | 36.0 | 124 |

正解選択肢は、不正解選択肢より平均で約 3 語長くなっています。これは軽いデータ生成上の癖として利用される可能性があります。

選択肢ごとの平均語数は `A`: 26.1、`B`: 26.4、`C`: 26.4、`D`: 25.9、`E`: 26.1 で、選択肢ラベル自体には大きな長さ差はありません。

## 長さバイアス

各設問内で「語数が長い選択肢」を上位に置く単純ルールを試すと、train 上の MAP@3 は 0.5375 でした。逆に「短い選択肢」を上位に置くと 0.2508 です。

正解選択肢の長さ順位も、長い側に寄っています。

| 正解選択肢の長さ順位（長い順） | 件数 |
|---|---:|
| 1位 | 73 |
| 2位 | 49 |
| 3位 | 30 |
| 4位 | 22 |
| 5位 | 26 |

このバイアスは非常に重要です。ただし、隠しテストで同じ癖が残る保証はありません。長さだけのルールに寄せるより、モデル出力や検索スコアと組み合わせる補助特徴として扱うのが安全です。

## 重複・異常値

- `prompt` の重複は 0 件です。
- 同一設問内で完全一致する選択肢はありません。
- 全選択肢テキスト 1,000 セルのうち、完全重複は 1 件だけです。
  - `id=101`, `D`: `None of the above.`
  - `id=102`, `A`: `None of the above.`

全体として、CSV の欠損や明らかな壊れは見当たりません。

## EDA から見える実験方針

1. **検証セットを必ず自作する**
   - 配布 `test.csv` は `train.csv` と同一なので、性能評価には使えません。
   - 200 件しかないため、単純 holdout だけでなく cross-validation も検討したいです。

2. **強い baseline を先に作る**
   - 固定順序 `B C D` で MAP@3 0.4133。
   - 長さ順ルールで MAP@3 0.5375。
   - まずはこれらを超えることを最低ラインにします。

3. **multiple choice 分類として扱う**
   - 入力は `prompt + option` のペアを 5 個作り、各選択肢の正しさをスコア化する形式が自然です。
   - DeBERTa/BERT 系の `AutoModelForMultipleChoice` または cross-encoder 型の分類が合いそうです。

4. **RAG を重視する**
   - 設問だけでは短く曖昧なものが多く、関連文脈の検索が効きやすい構造です。
   - Wikipedia や科学 QA データを検索し、文脈付きで選択肢を評価する方針が有望です。

5. **長さ特徴は補助的に使う**
   - 正解が長い傾向は train では明確です。
   - ただし過学習しやすいため、モデルスコアの tie-breaker や特徴量の一部として扱うのが良さそうです。
